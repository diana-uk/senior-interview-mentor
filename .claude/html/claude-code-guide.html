Claude Code: Global & Project Skills

How to set up skills, memory, and configuration at different scopes

The Two-Layer System

Claude Code uses a hierarchical configuration system. Anything you put in the global directory (~/.claude/) applies to every project on your machine. Anything inside a project's .claude/ folder applies only to that repo. Both layers load automatically and merge together.

Global ~/.claude/ → Project .claude/ → Merged result
Override rule: When both layers define the same thing (e.g. a skill named /review), the project-level version wins.

Global Global Configuration

Your global config lives in your home directory. These skills, memory, and settings are available in every project you open with Claude Code.

Global Directory Structure
~/.claude/
├── CLAUDE.md              # Global instructions & preferences
├── settings.json          # Global permissions, env vars
├── mcp.json               # Global MCP servers
├── skills/                # Skills available EVERYWHERE
│   ├── explain-code/
│   │   └── SKILL.md
│   └── research/
│       └── SKILL.md
├── agents/                # Global subagents
└── rules/                 # Global rules
    └── my-preferences.md

Global CLAUDE.md

Put personal preferences that apply everywhere in ~/.claude/CLAUDE.md:

# My Preferences

- Always use TypeScript unless I say otherwise
- Use 2-space indentation
- Prefer functional patterns over classes
- When committing, use conventional commit format

Global Skill Example

Create a folder inside ~/.claude/skills/ with a SKILL.md file:

# File: ~/.claude/skills/explain-code/SKILL.md

---
name: explain-code
description: Explains code with diagrams and analogies
---

When explaining code:
1. Start with an everyday analogy
2. Draw an ASCII diagram of the flow
3. Walk through step-by-step
4. Highlight a common gotcha

Now /explain-code works in every project on your machine.

Project Project Configuration

Project config lives in your repo root. Skills here only apply to this project and can be committed to git so your team gets them too.

Project Directory Structure
your-project/
├── CLAUDE.md              # Project instructions (committed)
├── .claude/
│   ├── settings.json      # Project permissions
│   ├── settings.local.json  # Personal overrides (auto-gitignored)
│   ├── CLAUDE.local.md    # Personal project notes (auto-gitignored)
│   ├── skills/            # Project-only skills
│   │   ├── deploy/
│   │   │   └── SKILL.md
│   │   └── run-tests/
│   │       └── SKILL.md
│   ├── agents/            # Project subagents
│   └── rules/             # Project rules (modular)
│       ├── code-style.md
│       └── testing.md
└── .mcp.json              # Project MCP servers

Project Skill Example

# File: .claude/skills/deploy/SKILL.md

---
name: deploy
description: Deploy the app to production
---

Deploy steps:
1. Run the full test suite
2. Build the application
3. Push to the deployment target
4. Verify the deployment succeeded
Personal files: settings.local.json and CLAUDE.local.md are automatically gitignored. Use these for personal preferences that shouldn't be shared with your team.

Anatomy of a SKILL.md

Every skill is a folder containing a SKILL.md file. The file has two parts:

Frontmatter (YAML)

Required metadata between --- fences.

  • name — the slash command name
  • description — when/why to use it
Body (Markdown)

Instructions Claude follows when the skill is invoked.

  • Step-by-step instructions
  • Templates, rules, examples
---
name: my-skill
description: Short description of what this does
---

## Instructions

Your markdown instructions here.
Claude follows these when the user runs /my-skill.

Supporting Files

Skills can include extra files alongside SKILL.md:

~/.claude/skills/my-skill/
├── SKILL.md
├── template.md
├── examples/
│   └── sample.ts
└── scripts/
    └── helper.sh

Loading & Priority Order

When Claude Code starts, configuration layers load in this order (later wins):

1. Managed → 2. Global ~/.claude/ → 3. Project .claude/ → 4. Local overrides
Layer Source Shared?
Managed Enterprise IT deployment Org-wide
Global ~/.claude/ Your machine only
Project .claude/ in repo root Committed to git (team)
Local .claude/*.local.* Auto-gitignored (you only)

Quick Reference Table

What Global Path Project Path
Skills ~/.claude/skills/<name>/SKILL.md .claude/skills/<name>/SKILL.md
Memory ~/.claude/CLAUDE.md ./CLAUDE.md
Personal notes — .claude/CLAUDE.local.md
Settings ~/.claude/settings.json .claude/settings.json
Personal settings — .claude/settings.local.json
MCP servers ~/.claude/mcp.json .mcp.json
Rules ~/.claude/rules/*.md .claude/rules/*.md

Tips

  • When you type /, Claude shows all available skills merged from both global and project layers.
  • Use /memory to check which CLAUDE.md files are currently loaded.
  • Skills can reference each other — a project skill can build on patterns from a global one.
  • Keep global skills generic (explain, research, format). Keep project skills specific (deploy, test, lint).
  • The older .claude/commands/ format still works but .claude/skills/ is the modern approach with more features.
