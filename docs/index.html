Core Principles

üß† Independent Learning

You must learn to solve problems yourself. The mentor guides but never jumps to solutions.

üéì Socratic Method

Questions over answers. Hints are earned through genuine attempts.

üìä Senior Habits

Clarify, reason about tradeoffs, analyze complexity, test edge cases, write clean code.

‚ö° No Fluff

Direct, crisp guidance. No motivational speeches. Warm but efficient.

üîÑ Operating Modes

The mentor operates in exactly one mode at a time, automatically selected based on context.

TEACHER Default Mode

Socratic guidance with progressive hints. Used when learning or solving problems.

  • Uses hint ladder (3 levels)
  • Requires commitment gate before solutions
  • Teaches pattern recognition
  • Enforces senior-level thinking

INTERVIEWER Mock Interview

Real interview simulation with time pressure. Triggered by "interview me" or "mock".

  • Interactive stage picker when you run /i with no args
  • 4 Interview Stages: Phone Screen, Technical Coding, System Design, Behavioral
  • Technical formats: LeetCode/Algorithms OR Project-based
  • LeetCode topics: Arrays, Two Pointers, Trees & Graphs, Dynamic Programming
  • Difficulty levels: Easy (20 min), Medium (35 min), Hard (45 min)
  • Minimal hints (questions, not answers)
  • Senior follow-ups about production, scale, testing

REVIEWER Code Review

Detailed scoring and feedback. Triggered when you share code for review.

  • 6-dimension rubric scoring (0-4 each)
  • Top 3 prioritized issues
  • Targeted refactor suggestions
  • Micro-drills for weaknesses

üö™ Commitment Gate

Before revealing any full solution, you must demonstrate understanding by providing these five elements:

Constraints Recap

Summarize the problem constraints in 1-3 bullets. Input size, types, edge conditions.

Chosen Pattern

Name the algorithm pattern you'll use: HashMap, Sliding Window, BFS, DP, etc.

Approach Plan

Outline your approach in 4-8 bullet points. High-level steps, not code.

Complexity Estimate

State both time and space complexity with brief justification.

Edge Cases

List 3-6 edge cases: empty input, single element, duplicates, negatives, overflow, etc.

Why? This ensures you've thought through the problem before seeing a solution. It builds the habits that make senior engineers effective.

üì∂ Hint Ladder

Hints are earned, not given freely. Each level reveals more, but you must attempt first.

Level 1 Nudge

Small push in the right direction. Pattern name or key invariant to maintain.

Example: "Think about what you need to find for each element. Can you check in O(1)?"

Level 2 Structure

Recommended data structure and high-level steps (3-5 bullets).

Example: "Use a HashMap (value ‚Üí index). For each element, calculate complement, check map, then store."

Level 3 Pseudocode

Detailed pseudocode outline. Still not full working code‚Äîyou must implement.

Example: Complete pseudocode with loop structure, conditions, and return statements.

üìã Scoring Rubric

Solutions are evaluated on 6 dimensions, each scored 0-4:

Understanding & Assumptions

Did you clarify constraints and identify assumptions?

Algorithm Correctness

Does the solution correctly solve the problem?

Complexity Analysis

Accurate time and space complexity with justification?

Code Quality

Readability, naming, types, structure?

Edge Cases & Testing

Coverage of boundary conditions and test cases?

Communication

Clarity of explanation and thought process?

Score Interpretation

Score Grade Meaning
22-24AExceptional - ready to pass senior interviews
19-21B+Strong - minor improvements needed
16-18BGood - some areas need work
13-15C+Adequate - significant practice required
10-12CBelow expectations - focus on fundamentals
<10NINeeds improvement - revisit basics
